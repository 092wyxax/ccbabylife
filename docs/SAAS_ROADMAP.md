# SaaS 化路線圖

> M1 已決議多租戶 schema，本文件規劃從「自家選物店」演化為「日系代購商家 SaaS 工具」的路徑。
> 預估 Phase 6 起動工，但前 5 個 Phase 的設計已預留多租戶切割。

---

## 1. SaaS 假設

- **目標客戶**：1–10 人規模的日本代購商家、選物店、IG 個體戶
- **價值主張**：
  1. 不用自己刻訂單系統 / 商品管理 / 物流整合
  2. 內建市場情報（拿來找選品 / 監競品）
  3. 內建 LINE Bot 自動報價（差異化最大）
  4. 不用懂工程

- **不打算做**：
  - 「全功能電商平台」（Shopify 已存在）
  - 「全品類市集」（蝦皮、Pinkoi 已存在）
  - 「自動 Dropshipping」（不符合誠信宣告）

## 2. 收費模型（粗估）

| 方案 | 月費 | 限制 | 適合誰 |
|---|---|---|---|
| 起步 | NT$1,200 | 50 SKU、100 訂單/月、無情報 | 個體戶 |
| 標準 | NT$3,500 | 200 SKU、500 訂單/月、含情報 | 1–3 人小店 |
| 進階 | NT$8,000 | 無限 SKU、2,000 訂單/月、含情報 + 分潤 + AI 客服 | 3–10 人 |
| 企業 | 議價 | 自訂 | — |

> 若採用：本店本身是「進階 + 自家定制」，不付月費（dogfood）。

## 3. 多租戶切分（M1 已預備）

### 3.1 資料層
所有業務表 `org_id` 已內建，RLS 策略：

```sql
CREATE POLICY "org isolation"
ON {{table}} FOR ALL
USING (org_id = (current_setting('request.jwt.claims', true)::json ->> 'org_id')::uuid);
```

### 3.2 路徑層
- 自家：`https://your-domain.com`（org_id = 預設 org）
- 客戶：`https://{{org-slug}}.platform-domain.com`（依 org slug 解析）
  - 或：`https://platform-domain.com/{{org-slug}}`（path 模式，先實作這個簡單）

### 3.3 配置層
每個 org 可自訂：
- 品牌（logo、色票、名稱）
- 定價公式（pricing_formula 表）
- LINE 官方帳號（每 org 一個）
- 綠界商家（每 org 一個）
- Email 寄件地址

### 3.4 共用層
所有 org 共用：
- 市場情報資料（爬蟲結果，分匿名 / 訂閱可見）
- AI Pipeline（Claude / OpenAI key 由平台支付，依用量計入帳單）
- 系統公告

## 4. 演化階段

### Stage 0：本店穩定營運（Phase 1–4 結束時）
- 自家本店持續運營 6–12 個月
- 累積真實營運經驗、bug 都修過
- 確認哪些功能 SaaS 客戶會買單、哪些不重要

### Stage 1：第一個外部客戶（試水）
- 找朋友圈內已在做日系代購的人，免費試用 3 個月
- 收回饋、修 bug
- 建立 onboarding SOP

### Stage 2：付費 alpha（5–10 個客戶）
- 開始收費，但不公開廣告
- 高觸控客服（先生 / 太太親自帶）
- 收集成功案例 / 證言

### Stage 3：公開 beta（30–50 個客戶）
- 加 self-serve onboarding
- 加部署自動化（新客戶 1 小時內可用）
- 加客戶端 dashboard 自助管理

### Stage 4：規模化（100+ 客戶）
- 加 partner 計畫
- 加 reseller 抽成
- 評估開發新模組（例如：日本帳本、批次報關）

## 5. 預期挑戰

### 5.1 技術
- 跨 org 資料隔離 bug 一旦發生 = 災難
  → Stage 1 起做 audit log + 自動化測試覆蓋 RLS
- AI 成本隨客戶數線性上升
  → 收費要 cover 成本，並考慮按用量計費
- 多 LINE 官方帳號管理複雜
  → 一開始要求客戶自己有 LINE 官方帳號，平台僅整合不代理

### 5.2 商業
- 客戶自己有粉絲了，為什麼用我們？
  → 答：省工程資源、市場情報拿來找選品、LINE Bot 自動報價
- 客戶可能 churn（自己做大了 / 不做了）
  → 答：年費鎖客 / 提供多元留存機制
- 法規誠信宣告對 SaaS 客戶的約束力？
  → 答：條款規定不接紅燈品項，違者剔除

### 5.3 法律
- 客戶資料（其客戶的個資）由我們儲存，責任歸誰？
  → 簽 DPA（Data Processing Agreement），明確雙方責任
- 客戶間的競爭（同樣賣 Pigeon 奶瓶的兩家都在我們上）
  → 不限制（這是市場行為），但情報資料嚴格隔離

## 6. Phase 6 動工先決條件

不可以為了 SaaS 而 SaaS。動工前確認：

- [ ] 本店月營收連續 6 個月穩定 > NT$300,000
- [ ] 本店毛利率穩定 > 25%
- [ ] 已有 3+ 同業朋友主動詢問「能不能用你們的系統」
- [ ] 三人共識（PARTNERSHIP §4.3）
- [ ] 有財力 / 體力多承擔此分支

若上述條件未滿足 → Phase 6 改為「持續優化本店」，SaaS 暫不啟動。
